## Critical Findings

### 1. `vibe-vocab` — Database URL exposed via VITE_ prefix (**PUBLIC REPO**)

**File:** `src/lib/db.ts`  
**Evidence:** `process.env.DATABASE_URL || process.env.VITE_DATABASE_URL`  
**CWE:** CWE-200 (Information Exposure)  
**OWASP:** A01:2021 Broken Access Control  

The `VITE_` prefix causes Vite to bundle this value into the client JavaScript bundle. Anyone opening DevTools → Sources can see the full Postgres connection string (host, username, password, database name). This is a **public repo**.

**Fix:** Remove the `VITE_DATABASE_URL` fallback. Database URLs must only be accessed server-side via `process.env.DATABASE_URL` (no VITE_ prefix). The Neon serverless driver should only run in API routes/server functions, never client-side.

---

### 2. `chromatic-illusion-weaponizer` — Password exposed to client bundle

**File:** `.env.example` declares `VITE_LOGIN_PASSWORD`  
**CWE:** CWE-798 (Use of Hard-coded Credentials)  
**OWASP:** A07:2021 Identification and Authentication Failures  

The `VITE_` prefix means this password is embedded in the client-side JavaScript bundle. Anyone can extract it from DevTools → Sources or the network response. Client-side password checks provide zero security.

**Fix:** Move authentication to the server. Use a proper auth flow (session cookies, JWTs from server) instead of comparing passwords in the browser.

Any website can make authenticated API calls to these endpoints.

### 4. `ruby-feynman` — XSS via dangerouslySetInnerHTML

**File:** `src/App.jsx`  
**Evidence:** `dangerouslySetInnerHTML={{ __html: formatInline(content) }}`  
**CWE:** CWE-79 (Cross-site Scripting)  

Content is rendered as raw HTML. If `formatInline()` doesn't sanitize, any `<script>` tag or event handler in the input will execute.

**Fix:** Use DOMPurify to sanitize: `dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(formatInline(content)) }}`

---

### 5. `digital-alchemy-app` — XSS via unsanitized HTML rendering

**File:** `webapp/src/pages/Resources.tsx`  
**Evidence:** `dangerouslySetInnerHTML` with raw regex-based formatting (`.replace(/\*\*(.*?)\*\*/g, ...)`)  
**CWE:** CWE-79 (Cross-site Scripting)  

Paragraph content is converted to HTML via regex substitution and injected unsanitized.

**Fix:** Sanitize with DOMPurify before rendering, or use a Markdown library with XSS protection.

---

### 6. `digitalalchemy-dev` — dangerouslySetInnerHTML in admin panels

**Files:** `src/app/admin/freebies/CampaignCockpitClient.tsx`, `src/app/admin/newsletter/NewsletterCampaignComposer.tsx`  
**Evidence:** `dangerouslySetInnerHTML={{ __html: compileLivePreview(bodyHtml) }}`  

Admin-authored HTML rendered without visible sanitization. If `compileLivePreview` doesn't sanitize, this is an XSS vector (even in admin panels, stored XSS can escalate).

**Fix:** Pass through DOMPurify or use a sandboxed iframe for previews.

---

### 7. `1code-main` — Multiple dangerouslySetInnerHTML usages

**Files:** `agent-tool-call.tsx` (2 locations), `chat-markdown-renderer.tsx`, `agent-edit-tool.tsx` (2 locations)  
**Evidence:** `dangerouslySetInnerHTML={{ __html: subtitleStr }}` and `{{ __html: highlightedHtml }}`  

These render tool-call subtitles and syntax-highlighted code as raw HTML. If the source data isn't fully trusted and sanitized, this creates XSS risk.

**Fix:** Ensure all values passed to `__html` go through a sanitizer (DOMPurify). The `highlightedHtml` case is likely safe if generated by Shiki/Prism, but `subtitleStr` should be verified.

### 8. `digitalalchemy-dev` — Console logging of token metadata

**File:** `scripts/google-token-probe.ts`  
**Evidence:** `console.log(\`refresh token: ${data.refresh_token ? "present" : "MISSING"}\`)`  
**Severity:** Low (script-only, not production code)  

Logs token status to console. The script doesn't log the actual token value, only its presence — acceptable for a diagnostic script but should be removed before any production use.
